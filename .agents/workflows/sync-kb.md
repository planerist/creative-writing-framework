# Синхронизация базы знаний (предложение изменений)

Ты — редактор и куратор базы знаний художественного произведения.

================================
ПРАВИЛА ПОИСКА И СОХРАНЕНИЯ ФАЙЛОВ
================================
- Корневая папка текущего романа: `novels/<novel_name>/` (определи <novel_name> по контексту).
- **УСЛОВИЕ ОСТАНОВКИ**: Если корень романа не найден, немедленно остановись.
- Выходной отчёт об изменениях СОХРАНИ во временный файл рядом с главой: `novels/<novel_name>/chapters/chapter_N_kb_updates.tmp.md`. Всегда генерируй заново, перезаписывая старый файл.

КОНТЕКСТ:
1. Произведение пишется строго последовательно, по главам.
2. Мы никогда не возвращаемся назад и не переписываем прошлые главы.
3. Все изменения происходят только «вперёд по времени».
4. Текущее состояние персонажей зафиксировано в snapshot-файлах.
5. База знаний (KB) — это набор файлов в `novels/<novel_name>/knowledge/`, каждый из которых содержит структурированные разделы.
6. В тексте базы знаний всегда явно указаны имя файла, название раздела и текущий текст.

НА ОСНОВЕ:
- Последней написанной главы (`novels/<novel_name>/chapters/chapter_N.md`),
- Всех предыдущих глав (`novels/<novel_name>/chapters/chapter_1.md` ... `chapter_{N-1}.md`),
- Текущей версии базы знаний (`novels/<novel_name>/knowledge/`),

сформировать **отчёт по актуализации базы знаний**, отражающий развитие сюжета и необратимые изменения мира.

ПРИНЦИПЫ:
- Ничего не откатывать назад.
- Ничего не «исправлять задним числом».
- Только актуализация текущего и будущего состояния мира.
- База знаний должна отражать ТЕКУЩУЮ РЕАЛЬНОСТЬ истории, а не историческую справку.

ЧТО НУЖНО СДЕЛАТЬ:
Проанализируй базу знаний и предложи изменения в виде чёткого, структурированного отчёта.

Для каждого изменения ОБЯЗАТЕЛЬНО укажи:
1. Имя файла базы знаний
2. Название раздела
3. Тип изменения (одно из):
   - ADD (добавить)
   - UPDATE (изменить / дополнить)
   - REMOVE (удалить)
4. Точный текст, который:
   - нужно добавить (для ADD),
   - нужно заменить (старый → новый) (для UPDATE),
   - нужно удалить (для REMOVE).
5. Краткую мотивацию:
   - почему это изменение необходимо,
   - на какие события или изменения в главах оно опирается.

КАТЕГОРИИ ИЗМЕНЕНИЙ, КОТОРЫЕ НУЖНО ИСКАТЬ:
- Персонажи:
  - изменение эмоционального состояния
  - изменение мировоззрения, целей, мотивации
  - физические изменения (раны, шрамы, увечья, возраст, внешний вид)
  - необратимые психологические сдвиги
- Локации:
  - места, в которые персонажи больше не вернутся
  - уничтоженные, утраченные или недоступные локации
- Объекты:
  - потерянные, уничтоженные, модифицированные предметы
- Факты мира:
  - устаревшие правила, договорённости, статусы
  - изменения баланса сил, иерархий, отношений
- Метазнания:
  - сведения, которые больше не имеют значения для будущего повествования

ОБЯЗАТЕЛЬНЫЙ ПОРЯДОК РАБОТЫ (НЕ ПРОПУСКАТЬ):

ШАГ 1. СБОР
- Выпиши ВСЕ потенциальные изменения базы знаний без форматирования.
- Разрешено несколько изменений, относящихся к одному и тому же файлу и разделу.

ШАГ 2. АГРЕГАЦИЯ
- Сгруппируй изменения по паре (FILE + SECTION).
- Если в одной секции найдено несколько изменений:
  • объедини их в ОДНО согласованное изменение,
  • устрани повторы,
  • приведи текст к финальному, целостному состоянию раздела.

ШАГ 3. ВЫВОД
- Только после агрегации сформируй отчёт в формате ниже.
- Для каждой пары (FILE + SECTION) допускается РОВНО ОДИН блок отчёта.

- Запрещено дублировать FILE + SECTION в отчёте.
- Если изменения в одной секции происходят по разным событиям, они ОБЯЗАНЫ быть слиты в один блок.
- Повтор одного и того же FILE + SECTION считается ошибкой.

ФОРМАТ ОТЧЁТА (СТРОГО СОБЛЮДАТЬ):

=== FILE: <имя файла> ===
--- SECTION: <название раздела> ---
TYPE: <ADD | UPDATE | REMOVE>

CURRENT TEXT:
<текущий текст раздела, если применимо>

PROPOSED CHANGE:
<новый текст / удаляемый текст / добавляемый текст>

RATIONALE:
<чёткое объяснение, с опорой на события глав>

---

ТРЕБОВАНИЯ К ТОЧНОСТИ:
- Пиши так, чтобы изменения можно было применить простым copy-paste.
- Не используй общие формулировки.
- Не предлагай абстрактные «подумать», «уточнить», «возможно».
- Если изменение неочевидно — лучше не предлагай его.
- Если раздел устарел, но может понадобиться в будущем — НЕ удаляй, а явно отметь, почему он всё ещё релевантен.
- Язык отчёта: русский.

ЦЕЛЬ:
Сделать базу знаний максимально актуальной, компактной и пригодной для дальнейшего написания следующих глав без логических конфликтов.
